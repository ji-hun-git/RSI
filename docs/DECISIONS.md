# Decision records

ADR-style records for the places where this Stage-1 implementation deviates from, or narrows, the recommendations in `TECHNICAL_REPORT.md`. Schema changes to `foundry.contracts` require a new record here (see `CLAUDE.md`: contracts are frozen).

## ADR-001: Single installable package instead of the report 17.1 multi-package monorepo

- **Context.** Report 17.1 proposes a monorepo with separate `packages/` (contracts, event-ledger, policy, registry, ...) plus `apps/` and `adapters/`. Stage 1 is one small reference implementation maintained by one team.
- **Decision.** Ship one installable distribution (`agent-foundry`) with one import root (`foundry`), keeping the report's module boundaries as subpackages (`foundry.contracts`, `foundry.ledger`, `foundry.policy`, ...) that depend on each other only through the structural protocols in `contracts/protocols.py`.
- **Consequences.** One `pip install -e .`, one test run, no cross-package versioning overhead. The boundaries are preserved logically, so splitting into real packages later is a mechanical move. The top-level `adapters/`, `modules/`, `bundles/`, `benchmarks/`, `policies/` and `research/` directories from 17.1 already exist as data/adapter homes.

## ADR-002: SQLite plus file store instead of Postgres/S3

- **Context.** Report 16.1 recommends PostgreSQL + pgvector and an S3-compatible object store. Stage 1 must run locally on Windows and POSIX with zero services, and `foundry verify` must audit a root directory as a self-contained evidence package.
- **Decision.** `EventLedger` is a single-file SQLite database (append-only schema, hash chain, signed tip checkpoint); `ArtifactStore` and `BundleRegistry` are content-addressed files under the same root.
- **Consequences.** A foundry root is portable, diffable and auditable offline. Consumers depend on `LedgerLike` and `ArtifactStoreLike`, not on SQLite or the filesystem, so a Postgres/S3 backend is a drop-in replacement behind the same protocols. Concurrency is deliberately modest (BEGIN IMMEDIATE serialization), which is sufficient for Stage-1 workloads.

## ADR-003: HMAC dev signing as the Sigstore stand-in

- **Context.** The report calls for Cosign/Sigstore-grade signing of bundles and governance records (16.1, 14.5). Stage 1 needs the full sign/verify workflow exercised without external infrastructure.
- **Decision.** `HMACSigner` (HMAC-SHA256, key in a local file) signs events, bundles, promotion decisions and ledger checkpoints. Verifiers load keys with `HMACSigner.load` and never mint keys into a root under audit; a missing key is reported as its own outcome, never as forgery.
- **Consequences.** Integrity and signer identity are real; non-repudiation is not (a symmetric key can both sign and verify). Every trust decision flows through `registry/signing.py` and the `SignerLike` duck types, so swapping in asymmetric signatures changes one module.

## ADR-004: Deterministic fixture worker and slugify corpus instead of LLM workers

- **Context.** Report 18.4 Phase A validates infrastructure on deterministic fixture missions before any model runs; the Stage-1 exit questions (event completeness, replay, fair comparison) are unanswerable with a stochastic worker in the loop.
- **Decision.** `FixtureWorker` is a pure function of `(task_input, config, seed)` whose `strategy` config selects a naive or robust slugify transformation; `generate_task_sets(seed)` produces the four-role corpus, with `robust_slugify` as the single source of ground truth for both the oracle and the robust strategy.
- **Consequences.** Candidate-vs-control experiments have a known answer, replay is bit-exact, and `foundry verify` can recompute the full paired analysis from seeds alone. `WorkerLike` is the seam: OpenHands and mini-SWE-agent integrate behind the identical `invoke` contract without touching the control plane.

## ADR-005: Injected callables (RunArm/Score) decouple the experiment controller from the runtime

- **Context.** The experiment controller must compare arbitrary workers under arbitrary bundles without importing any runtime, and the vault must be able to execute candidates on protected tasks without revealing ground truth.
- **Decision.** `ExperimentController.run` takes `run_arm: (bundle, task, seed) -> output` and `score: (task, output) -> float` as arguments; the protected role is executed only through `HoldoutVault.run_blind`, which wraps `run_arm` in a redacted view and applies the seal-time scorer internally.
- **Consequences.** The controller is runtime-agnostic and trivially testable; scoring for open roles is caller-supplied while protected scoring is vault-owned, which is exactly the trust split report 14.1 requires. The same callables are reused by `verify`'s independent re-analysis path.

## ADR-006: Python 3.11 floor

- **Context.** Report 16.1 recommends Python 3.12+. The development and CI machines in scope for Stage 1 include 3.11 interpreters, and nothing in the codebase requires 3.12 features.
- **Decision.** `requires-python = ">=3.11"` with ruff targeting `py311`; CI exercises 3.11 and 3.12 on Linux and Windows.
- **Consequences.** Broader machine reality coverage at zero feature cost. Corpus generation and bootstrap analysis draw randomness only through `random.Random.random()` (the one generator method with a documented cross-version stability guarantee), so results are identical across interpreter versions. Revisit the floor when a 3.12-only feature earns its keep.

## ADR-007: Adapter code lives in src/foundry/adapters/ behind optional dependency groups

- **Context.** Report 17.1 places adapters in a top-level `adapters/` tree, which in the multi-package monorepo would be independently installable packages. ADR-001 collapsed Stage 1 into a single installable package, so a separately packaged adapter tree would reintroduce exactly the packaging overhead ADR-001 removed, while an adapter inside the core dependency set would force LangGraph onto every user of the dependency-free core.
- **Decision.** Adapter implementations live in `src/foundry/adapters/` (e.g. `langgraph_runtime.py`) guarded by optional dependency groups in `pyproject.toml` (`pip install agent-foundry[langgraph]`); importing the module without the extra raises with the install hint. The top-level `adapters/` directories remain the per-adapter documentation and the report-layout anchor. Shared runtime behavior was extracted to `foundry.runtime.LedgerBackedRuntime` and `foundry.runtime.fixture_workflow` so adapters supply scheduling only, never record semantics, and `tests/test_runtime_conformance.py` runs the full RuntimeAdapter behavior suite against every installed runtime plus a byte-equivalence check against the deterministic reference.
- **Consequences.** The core stays dependency-light and the canonical event stream is runtime-invariant by construction, which is the report 9.3 requirement stated as a test. When an adapter graduates to its own distribution (Stage 2+), the module moves out with its optional group and the conformance suite comes along unchanged.

## ADR-008: Executable-test coding domain with a trusted-checks test service, before any model-backed coding agent

- **Context.** Report 18.3 makes "small software tasks with executable tests" the primary first domain, and HANDOFF queued OpenHands/mini-SWE-agent workers next. But real agent runs need LLM keys and a real sandbox, neither of which exists in Stage 1, and the report's own known failure mode "edits tests to fit incorrect implementation" (Appendix A) plus threat "tamper with tests" (14.2) must be answered by infrastructure, not by trusting the worker.
- **Decision.** Ship the domain substrate first, fully deterministic: `foundry.workers.coding_tasks` (tiny buggy repos + assertion scripts, four roles, generated by construction from seed and index), `DeterministicCodingWorker` (pure repair strategies; robust reproduces the hidden ground truth exactly), and `foundry.evaluation.DeterministicTestService`, which materializes candidate output in an ephemeral workspace, force-restores the trusted checks file, executes with a hard timeout and emits command receipts (14.4). Scoring untrusted output is fail-closed (malformed JSON, path traversal, timeout all score 0.0). Workers stay pure functions; only the test service executes anything.
- **Consequences.** Paired experiments now run on run-the-checks evidence with a known answer, and the test-tampering defense is a pinned test rather than an assumption. The local subprocess is documented as a determinism convenience, NOT a security sandbox: model-generated code must not execute here, so the OpenHands/mini-SWE adapters (whose `WorkerLike` seam and task shape this domain now specifies exactly) remain blocked on a real sandbox behind the same `run_checks` interface. That ordering is the report's own stage gating (19.6) applied to ourselves.

## ADR-009: Memory as an event-sourced projection with quarantine-first writes

- **Context.** Report 11 requires typed, governed memory: staged candidate writes, quarantine, provenance-required promotion, contradiction links, expiry and filters-before-match retrieval, with the 11.6 safety invariant that nothing becomes production memory merely by appearing in a successful trajectory. A separate memory database would create a second write path competing with the evidence root (the report's 16.2 warning against multiple writeable memory systems).
- **Decision.** `foundry.memory.MemoryService` stores no authoritative state of its own: every lifecycle change (candidate written, reviewed, promoted, contradicted, expired, retrieval feedback) is a canonical `memory.*` ledger event, and current state is the pure projection `project_memory(events)`, same pattern as the deployment controller. The 11.3 authority table is structural: GOVERNANCE items are refused (no autonomous path), PROCEDURE items are refused with a pointer to the bundle registry lifecycle, staging force-resets verification to `unverified` (review evidence cannot be self-declared), promotion requires review status plus real provenance plus a promoter distinct from the author (no self-promotion), and negative lessons additionally need an applicability scope and a reconsideration condition. Retrieval applies identity/security/read-scope/temporal filters BEFORE any relevance matching; contradicted and expired items leave retrieval but stay in history and surface as `ContextPackage` warnings via `ContextBuilder`, which also enforces an explicit token budget with an omitted-item count and emits MEMORY_SHOWN per included item.
- **Consequences.** Memory state is reproducible from the ledger alone (pinned by a second-service rebuild test), poisoning requires defeating review-and-promote rather than just writing, and projections stay disposable per 11.6. Open, deliberately: semantic extraction pipelines (models proposing candidates), consolidation jobs, vector/graph projections and the privacy deletion/redaction workflow; the staged-write API is the seam for all of them.

## ADR-010: Authority-free proposer seam with a deterministic template-mutation reference

- **Context.** Report 12.4 orders candidate-generation strategies by risk: human-designed diffs, then template mutation ("bounded edits to known manifest fields -- safest first automation"), then GEPA/DSPy/TextGrad. HANDOFF queued the GEPA/DSPy adapter, but a model-backed proposer needs an LLM key, and shipping the seam without a working reference would leave the interface unexercised and the authority boundary untested.
- **Decision.** `foundry.improvement` ships the loop's first three steps as deterministic code: `EvidenceDiagnoser` (strictly read-only; groups mission-level metric failures by (bundle, family, difficulty) with a support minimum, and refuses evidence it cannot attribute to a frozen bundle config) and the `ProposerLike` protocol with `TemplateMutationProposer` as reference (a governance-supplied mutation table instantiated against diagnosed failures; it cannot invent targets). Constraints are explicit inputs (`ProposalConstraints`: allowed mutation surface, budgets, pre-registered thresholds, rejected diffs), and two report 12.5 stopping conditions are code: the proposal budget caps output, and a diff whose digest matches a rejected candidate is re-emitted only with new evidence ids. Proposers hold no registry/vault/gate handles; the PDP denies `holdout.read` to optimizer principals.
- **Consequences.** The complete improvement loop (cohort -> diagnosis -> proposal -> policy-checked fork -> paired blind experiment -> gate) runs end-to-end with no model, pinned by `tests/test_improvement_loop.py`, so the GEPA/DSPy adapter reduces to implementing `ProposerLike` behind an API key with every downstream authority already enforced. Diagnosis quality (RQ1, causal accuracy) stays an open research question; the diagnoser only claims repeatable-signature grouping, not causal proof.

## ADR-011: Direct-API reflective proposer with governance-declared value domains, before GEPA/DSPy libraries

- **Context.** With an OpenAI key available, the queued GEPA/DSPy adapter became buildable. But both libraries are heavy dependencies whose value (evolutionary search, program compilation) exceeds what Stage 1 can evaluate; the report 12.4 mechanism they share is reflective proposal generation from trace evidence, which a direct API call implements in one small, auditable module. The first live run also produced a finding worth encoding: the model proposed strategy='advanced', a value that does not exist. Governance contained it (the candidate cannot execute; the parent survives), but legal values for closed-domain fields are governance information the proposer should receive and be validated against.
- **Decision.** `foundry.adapters.openai_proposer.OpenAIReflectiveProposer` implements `ProposerLike` over the OpenAI chat completions API (optional dependency group `openai`, lazily imported; any object with `chat.completions.create` injects for offline tests). Model output is untrusted data validated fail-closed: out-of-surface paths dropped, `old_value` read from the parent bundle (a hallucinated current state cannot enter the diff), no-ops dropped, out-of-domain values for `ProposalConstraints.value_domains` fields dropped, rejected-diff and budget rules identical to the template proposer, malformed output yields zero proposals. Every API call is ledgered as MODEL_REQUEST/MODEL_RESPONSE with provider, model, sampling settings, prompt/response digests and token usage, and those event ids join the proposal's evidence chain (report 12.3). The API key lives in the environment only, never in code, events, artifacts or logs; the live test skips without it.
- **Consequences.** The full improvement loop has run live under complete governance (diagnosis -> model proposal -> paired blind experiment -> quarantine without approval -> reject of model self-approval -> canary with human A1), with the model's participation itself auditable from the ledger. GEPA/DSPy library adapters remain open behind the identical seam for when evolutionary search over candidates is actually wanted; nothing downstream changes when they land.

## ADR-012: The registered campaign is committed evidence, and coverage is measured against a declared vocabulary

- **Context.** Two report 19.1 exit criteria remained open: "20+ paired candidate/control experiments reproducible" (unmet as a recorded research campaign) and "at least 95% required event coverage on fixtures" (no automated meter). A campaign that lives only on one machine is not evidence, and a coverage number is meaningless without saying coverage *of what*.
- **Decision.** `foundry.experiment.campaign` runs pre-registered, fixed-size, single-pass campaigns at full protocol standard (holdout sealed first, leakage check, matched seeded arms, paired bootstrap, G0-G9 gate WITHOUT approval -- a campaign measures, it does not deploy) and returns a deterministic results payload (no random ids, no wall-clock). The v1 design (12 slugify + 8 coding experiments, seeds 101-112/201-208) is pre-registered in `research/preregistrations/STAGE1_CAMPAIGN_V1.md`; `scripts/run_stage1_campaign.py` archives the results JSON, the full hash-chained event log (JSONL) and a human-readable report under `research/`. The archive is live evidence: CI recomputes sample rows from the registered seeds and compares bit-for-bit, re-verifies the archived event chain, and pins the pre-registration's content digest so post-hoc design edits are detectable. `foundry.evaluation.coverage` measures event coverage against declared vocabularies (`DEMO_REQUIRED_EVENTS` for state roots, `STAGE1_FIXTURE_REQUIRED_EVENTS` adding the interruption vocabulary the fixture suite exercises), exposed as `foundry coverage --root` with the 95% threshold as exit code.
- **Consequences.** All report 19.1 exit criteria are now met except the trace UI (CLI remains the observability surface). Campaign v1's six pre-registered predictions all held, including QUARANTINE in all 20 experiments: the strongest governance statement in the archive is that a campaign whose evidence passes every gate still deploys nothing, because authority is never implicit. Re-running the campaign after fixture or analysis changes requires a new pre-registration version, not an edit to v1.

## ADR-013: The trace UI is a self-contained static HTML projection, not a live server

- **Context.** Report 15.3 lists a dashboard of views and 15.6 names the foundry-specific "lineage/evolution/governance" view as the missing contribution; it is the last open Stage-1 exit-criterion item (19.1 "Basic trace UI"). Report 15.4 is emphatic that observability can be actively harmful, so the how matters as much as the what. A live web server would add a framework dependency, a runtime attack surface and a non-reproducible artifact, against the repo's dependency-light, evidence-first ethos.
- **Decision.** `foundry.dashboard` renders a single self-contained HTML file (inline CSS, no scripts, no external fetches, collapsibles via native `<details>`, theme-aware) in two pure stages: `build_dashboard_model` projects a ledger + registry into a frozen typed model (read-only, event-sourced, appends nothing), and `render_html` turns that model into HTML. `foundry dashboard --root DIR [--out FILE]` writes it. The renderer is bound by report 15.4 in code: every experiment delta carries its confidence interval (uncertainty never dropped); quarantined candidates, rolled-back deployments, failed gates, node failures, policy denials, leakage hits and in-flight-but-undecided proposals each get an explicit place (failed branches never omitted); every governance decision renders the exact typed diff and all G0-G9 gate results (no approval shown without its evidence); the header states the exact evidence snapshot (event count, ledger tip digest, chain-verification status); all dynamic strings are HTML-escaped and passed through a conservative secret redactor (15.5) that leaves `sha256:` evidence digests intact; and the copy frames each paired delta as matched-design evidence, never a general causal claim (15.4 "imply causality from correlation").
- **Consequences.** All report 19.1 Stage-1 exit criteria are now met. The page is diffable, archivable and safe to open offline; it verifies as well-formed HTML and its content is a deterministic function of (model, generated_at). Richer 15.3 views (a live Evidence Inspector over ContextBuilder, an Artifact Workshop over repository diffs) and a Phoenix/MLflow trace-workbench integration remain open for later stages behind the same read-only projection boundary.

### ADR-013 hardening (adversarial review)

A six-lens adversarial review (harmful-observability, escaping/injection, secret-leak, projection-correctness, determinism, spec-fidelity) surfaced defects that were triaged and fixed before the trace UI landed:

- All candidate arms are projected, not only the first: a losing candidate's evidence is a branch that must not be omitted (report 15.4).
- A missing or malformed confidence interval renders as "unavailable", never a fabricated zero-width `[0.0, 0.0]` that would read as certainty, and never a crash (the dashboard renders even on a broken chain).
- The evolution-tree lifecycle is derived from current deployment standing (active / rolled-back / rejected / quarantined / registered), not the cumulative canary set, so rolled-back bundles are labelled correctly.
- A corrupt bundle (content-address failure on load) surfaces as an incident instead of silently vanishing from the tree.
- Each governance decision renders an authoritative diff computed from the registry (parent vs candidate), so the exact change is shown even when the proposal link is absent.
- `reject` (a hard gate failure) and `quarantine` (evidence acceptable, authority absent) are distinct incident kinds and lifecycle labels, not folded together.
- The secret redactor (report 15.5) covers PEM/OpenSSH private-key blocks, `password`/`secret`/`api_key`/`token` assignments and session cookies, not just API-token prefixes.
- Evidence provenance (experiment id, plan ref, retention-set ref, evidence refs) is rendered, and the confidence level is described in the caption rather than asserted as a column label the payload cannot prove.

## ADR-014: Memory consolidation is a deterministic, quarantine-only producer

- **Context.** The governed `MemoryService` (ADR-009) had no producer: staging existed but nothing wrote candidates. Report 11.5 ("consolidation without premature universal rules") specifies how episodic evidence should become memory, and it is a genuine governance algorithm, not a convenience: cluster recurring episodes, state preconditions and counterevidence, search for disconfirming cases before creating a rule, require recurrence, promote narrowly, and retain rejects. A model-based extractor would add a dependency and reintroduce the exact "successful trajectory becomes a rule" failure the report warns against (11.6).
- **Decision.** `foundry.memory.MemoryConsolidator` is a deterministic, model-free consolidation job. It reads mission-level evaluation episodes (METRIC_COMPUTED with a mission id and task family; experiment-controller aggregate metrics are excluded), clusters them by (bundle config, family, difficulty) using only configurations recoverable from MISSION_STARTED (the diagnoser's attributability rule), and forms candidate patterns carrying support and counterevidence. A candidate is staged only if it meets the recurrence minimum AND its counterexample rate is within the allowed bound (default zero); within-bound counterexamples are attached as contradicting links, never hidden. Success clusters become semantic claims, failure clusters become negative lessons with an explicit reconsideration condition (11.3). Every candidate is STAGED through `MemoryService.stage` with full provenance (the supporting event ids) and a deterministic `memory_id` derived from the cluster signature, so the job is idempotent. The consolidator holds no authority beyond staging: it cannot promote, and its output is invisible to retrieval until a reviewer and a distinct non-author promoter act (the 11.6 invariant, enforced by the service).
- **Consequences.** The improvement loop now has a governed, auditable memory-learning arm end to end (a real naive-worker cohort yields "reliably succeeds on easy" and "reliably fails on hard", both quarantined). Confidence is reported as an observed recurrence frequency with that method named, never as truth. Refreshing a superseded candidate as a new versioned record when more episodes accrue, model-assisted extraction of finer claims, and cross-family generalization remain open behind the same staging API.

## ADR-015: Modules are admitted by conformance evidence, and replacements by shadow execution

- **Context.** Report 17 is emphatic that structural type-matching is not enough: "Replacement means more than matching an input and output type... The conformance suite is part of the connector" (17.3, the LEGO principle). A `SystemBundle` already pins modules by `module_id@version` in `module_refs`, but nothing resolved those refs or gated a module's admission on demonstrated behavior, so the "modular replacement" thesis (19.2) had no enforcement.
- **Decision.** `foundry.modules` implements the conformance-gated admission and hot-swap layer, model-free. `WorkerConformanceHarness` runs the `WorkerLike` conformance suite (determinism, statelessness, output shape, declared-input tolerance) and produces signable `ConformanceEvidence`; a check that cannot complete because the module raised is a failed check, never a harness crash. `ModuleRegistry.register` admits a module only when every check passes (quarantine-by-default: a non-conformant module is refused with the failing checks named), records the optionally signed evidence, enforces version immutability (a ref cannot be re-registered with a different manifest digest), and `resolve` binds a bundle's declared slot to an admitted implementation. `check_replacement` is the report 17.3 shadow-execution step: it runs old and new on identical inputs and reports byte-level agreement, so a swap is compatible only when behavior is preserved and a same-interface behavior change is reported case-by-case, never silently coerced.
- **Consequences.** Seeded incompatibilities (nondeterministic, stateful, wrong-shape, crashing workers) are detected and refused, and a genuine drop-in is distinguished from a behavior change (naive and robust workers both conform to `WorkerLike` yet the swap check flags exactly the cases where they diverge). This is the Stage-2 conformance spine, not Stage-2 completion: the registry is an in-process runtime registry (module implementations are Python objects, not serialized), the full 19.2 exit criterion (three worker modules and two tool providers hot-swapped end to end) and a tool-provider conformance suite remain open, and wiring the registry into the runtime's worker resolution (so missions resolve `module_refs` through it) is deliberately deferred to avoid rippling the runtime constructor before the seam is proven.

## ADR-016: The tool boundary is a capability-bound gateway with per-call receipts

- **Context.** Report 14.3 treats the tool boundary as a primary security surface: MCP and native tools must be terminated at a gateway that translates every call into an internal capability and receipt model, separates discovery from authorization, blocks SSRF targets, marks tool output as untrusted, and records a receipt per call (14.4). The `CapabilityToken` (short-lived, scoped, non-transferable) and the PDP existed, but nothing bound them at a tool call site.
- **Decision.** `foundry.tools.ToolGateway` is that termination point. Registration is discovery only (a TOOL_DISCOVERED event; presence in the catalog confers no permission). Every `call` is authorized by a capability token bound to the calling subject (deny-by-default: no capability, no call; non-transferable; expiry- and revocation-aware through an injectable authorizer), egress-checked against an SSRF policy that blocks loopback, private, link-local (cloud-metadata) and reserved addresses and enforces an optional domain allowlist, size-bounded (an oversized output is refused, not passed on), and receipted to the append-only ledger (TOOL_AUTHORIZED / TOOL_CALLED / TOOL_RESULT / TOOL_SIDE_EFFECT / TOOL_DENIED) with argument and response digests, capability id, latency and side-effect class. Output is returned flagged untrusted (data, not instructions). A repeated side-effecting call carrying the same idempotency key returns the recorded receipt without re-running the effect -- but the replay is checked for authorization first, so a cached result is never handed to an expired or revoked capability. The gateway authorizes with capabilities, not by routing through the promotion-policy PDP (capabilities are the report's tool-authorization model; the PDP governs the mutation surface, a different decision).
- **Consequences.** SSRF targets, unauthorized calls, transferred/expired capabilities, out-of-scope resources, oversized outputs and tool crashes are all refused fail-closed, each with a ledgered receipt. This uses only existing event types (no contract change). Honest scoping: the egress check vets the literal host and a production proxy must additionally resolve-and-pin DNS to defeat rebinding; wiring the gateway into the worker/runtime call path (so a mission's tool use flows through it) is a later integration, deferred to keep the seam proven before it is threaded through the runtime.
